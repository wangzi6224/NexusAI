## 3. 为什么 AI Chat 必须有会话系统

### 3.1 没有 Conversation 的系统是什么

没有 Conversation，后端只知道：

```text
这次用户问了什么
```

它不知道：

```text
这是哪个会话
之前聊过什么
这个用户正在讨论哪个主题
这轮回答要接着哪个上下文
```

这就是现在 `/chat` 的状态。

### 3.2 Conversation 是什么

Conversation 是 AI Chat 产品的“会话边界”。

前端类比：

```text
Conversation ≈ ChatGPT 左侧会话列表中的一项
Conversation ≈ IM 应用里的一个聊天窗口
Conversation ≈ 前端路由里的 /chat/:conversationId
```

后端类比：

```text
Conversation ≈ 一条业务主记录
Message ≈ 主记录下的明细记录
```

Node.js / NestJS 类比：

```text
ConversationService 管会话业务
MessageService 或 Store 管消息读写
Controller 只负责 HTTP 入参和出参
```

### 3.3 Message 是什么

Message 不只是“聊天记录”。

它有两个身份：

```text
1. 产品层：展示给用户看的聊天消息
2. 模型层：构造 LLM messages 的原材料
```

所以 Message 不能只保存：

```json
{
  "question": "...",
  "answer": "..."
}
```

而应该保存：

```json
{
  "id": "uuid",
  "conversation_id": "uuid",
  "role": "user",
  "content": "...",
  "metadata": {},
  "created_at": "..."
}
```

### 3.4 为什么要有 role

LLM API 不是简单传字符串，而是传结构化消息：

```python
[
    {"role": "system", "content": "你是一个专业的 AI 开发学习助手。"},
    {"role": "user", "content": "我正在学 FastAPI，记住我刚学到异常处理。"},
    {"role": "assistant", "content": "好的，你正在学习 FastAPI 的异常处理。"},
    {"role": "user", "content": "我刚才说我在学什么？"},
]
```

`role` 告诉模型这句话是谁说的：

| role        | 含义          | Week 3 是否使用     |
| ----------- | ----------- | --------------- |
| `system`    | 系统规则，定义助手行为 | 使用固定系统提示词       |
| `user`      | 用户输入        | 使用              |
| `assistant` | 模型历史回答      | 使用              |
| `tool`      | 工具调用结果      | Week 9 Agent 再用 |

如果没有 role，模型看到的是一坨拼接字符串，结构会变差，也不利于后续 Tool Calling 和 Agent Trace。

## 4. Conversation 和 Message 的核心概念

### 4.1 Conversation 字段设计

```json
{
  "id": "uuid",
  "title": "学习 FastAPI 异常处理",
  "summary": null,
  "model": "gemma4:e2b",
  "provider": "ollama",
  "status": "active",
  "created_at": "...",
  "updated_at": "..."
}
```

字段解释：

| 字段           | 是什么               | 为什么需要                         | 前端 / Node 类比          |
| ------------ | ----------------- | ----------------------------- | --------------------- |
| `id`         | 会话唯一 ID           | URL 和消息关联都靠它                  | 路由参数 `conversationId` |
| `title`      | 会话标题              | 左侧列表展示                        | ChatGPT 左侧标题          |
| `summary`    | 会话摘要              | Week 4 做长期记忆压缩                | Redux store 里的派生状态    |
| `model`      | 默认模型              | 不同会话可用不同模型                    | 用户选择模型配置              |
| `provider`   | 模型供应商             | 以后支持 OpenAI / Claude / Ollama | SDK provider name     |
| `status`     | active / archived | 预留归档能力                        | 软删除 / 归档状态            |
| `created_at` | 创建时间              | 排序、审计                         | `createdAt`           |
| `updated_at` | 更新时间              | 会话列表按最近更新排序                   | `updatedAt`           |

### 4.2 Message 字段设计

```json
{
  "id": "uuid",
  "conversation_id": "uuid",
  "role": "user",
  "content": "我正在学 FastAPI，记住我刚学到异常处理。",
  "metadata": {},
  "created_at": "..."
}
```

字段解释：

| 字段                | 是什么     | 为什么需要                                  |
| ----------------- | ------- | -------------------------------------- |
| `id`              | 消息唯一 ID | 前端渲染 key、后续 trace 关联                   |
| `conversation_id` | 所属会话 ID | 建立 Conversation 和 Message 的关系          |
| `role`            | 消息角色    | 让 LLM 理解谁说了什么                          |
| `content`         | 消息正文    | 模型上下文和前端展示都依赖它                         |
| `metadata`        | 附加信息    | 保存 model、provider、latency_ms、is_stream |
| `created_at`      | 创建时间    | 消息排序和调试                                |

### 4.3 为什么 id 用 UUID

第三周暂时没有数据库，不能依赖数据库自增 ID。

所以用：

```python
str(uuid.uuid4())
```

好处：

```text
本地 JSON 可直接生成唯一 ID
未来迁移数据库时可以映射 UUID PRIMARY KEY
前端可以直接用作 key
不同机器生成也不容易冲突
```

### 4.4 为什么时间用 ISO 字符串

用：

```python
datetime.now().isoformat()
```

原因：

```text
JSON 原生不支持 datetime 对象
ISO 字符串可读性好
前端 new Date(value) 容易解析
未来迁移数据库也容易转 timestamp
```

## 5. 逐行解释

### 5.1 `from typing import Any`

```python
from typing import Any
```

`metadata` 是一个灵活字段，里面可能保存：

```json
{
  "model": "gemma4:e2b",
  "provider": "ollama",
  "latency_ms": 1200,
  "is_stream": false
}
```

值可能是字符串、数字、布尔值、数组、对象，所以用 `Any`。

### 5.2 `ConversationCreateRequest`

```python
class ConversationCreateRequest(BaseModel):
```

表示创建会话的请求体。

```python
title: str = Field(..., min_length=1, description="会话标题")
```

* `...` 表示必填。
* `min_length=1` 防止空标题。
* `description` 会出现在 Swagger 文档里。

```python
model: str | None = Field(default=None, description="可选，指定该会话默认使用的模型")
```

如果不传 model，就使用 `.env` 里的默认 Ollama 模型。

### 5.3 `ConversationItem`

```python
class ConversationItem(BaseModel):
```

表示接口返回给前端的会话对象。

```python
id: str
```

会话唯一标识，后续前端要请求：

```text
/conversations/{conversation_id}/messages
```

```python
summary: str | None = None
```

第三周不做自动摘要，但先预留。Week 4 会把它用于 Summary Memory。

```python
message_count: int = 0
```

前端会话列表通常要显示消息数量，也方便调试。

### 5.4 `ConversationDetailResponse(ConversationItem)`

```python
class ConversationDetailResponse(ConversationItem):
    pass
```

现在详情和列表项字段一样，所以继承即可。

这相当于 TypeScript：

```ts
type ConversationDetailResponse = ConversationItem
```

但 Python 里用 class 继承表达。

### 5.5 `SendMessageRequest`

```python
content: str = Field(..., min_length=1, description="用户本次发送的消息内容")
```

`content` 是用户本轮输入。不能叫 `message`，因为后端已经有 Message 实体，`content` 更精确。

### 5.6 `MessageItem`

```python
metadata: dict[str, Any] = Field(default_factory=dict)
```

不要写成：

```python
metadata: dict = {}
```

原因：`{}` 是可变对象，在 Python 里作为默认值可能造成共享引用问题。`default_factory=dict` 每次创建新 dict，更安全。

```python
import json
import uuid
from datetime import datetime
from pathlib import Path
from typing import Any

from src.app.config import get_ollama_model
from src.app.exceptions import ConversationError
from src.app.paths import DATA_DIR

CONVERSATIONS_FILE = DATA_DIR / "conversations.json"
MESSAGES_FILE = DATA_DIR / "messages.json"

ALLOWED_MESSAGE_ROLES = {"system", "user", "assistant", "tool"}


def _now_iso() -> str:
    return datetime.now().isoformat()


def _ensure_json_file(path: Path) -> None:
    DATA_DIR.mkdir(parents=True, exist_ok=True)

    if not path.exists():
        path.write_text("[]", encoding="utf-8")


def _load_json_list(path: Path) -> list[dict[str, Any]]:
    _ensure_json_file(path)

    try:
        content = path.read_text(encoding="utf-8").strip()
        if not content:
            return []

        data = json.loads(content)

        if not isinstance(data, list):
            raise ConversationError(
                message="JSON 存储格式错误",
                detail=f"{path} 内容不是数组",
                status_code=500,
            )

        return data

    except json.JSONDecodeError as exc:
        raise ConversationError(
            message="读取 JSON 失败",
            detail=f"{path}: {exc}",
            status_code=500,
        ) from exc


def _save_json_list(path: Path, items: list[dict[str, Any]]) -> None:
    _ensure_json_file(path)

    try:
        path.write_text(
            json.dumps(items, ensure_ascii=False, indent=2),
            encoding="utf-8",
        )
    except OSError as exc:
        raise ConversationError(
            message="写入 JSON 失败",
            detail=f"{path}: {exc}",
            status_code=500,
        ) from exc


def load_conversations() -> list[dict[str, Any]]:
    return _load_json_list(CONVERSATIONS_FILE)


def save_conversations(conversations: list[dict[str, Any]]) -> None:
    _save_json_list(CONVERSATIONS_FILE, conversations)


def create_conversation(title: str, model: str | None = None) -> dict[str, Any]:
    now = _now_iso()

    conversation = {
        "id": str(uuid.uuid4()),
        "title": title,
        "summary": None,
        "model": model or get_ollama_model(),
        "provider": "ollama",
        "status": "active",
        "created_at": now,
        "updated_at": now,
    }

    conversations = load_conversations()
    conversations.append(conversation)
    save_conversations(conversations)

    return conversation


def list_conversations() -> list[dict[str, Any]]:
    conversations = load_conversations()
    return sorted(
        conversations,
        key=lambda item: item.get("updated_at", ""),
        reverse=True,
    )


def get_conversation(conversation_id: str) -> dict[str, Any] | None:
    conversations = load_conversations()

    for conversation in conversations:
        if conversation["id"] == conversation_id:
            return conversation

    return None


def update_conversation(
    conversation_id: str,
    updates: dict[str, Any],
) -> dict[str, Any]:
    conversations = load_conversations()

    for conversation in conversations:
        if conversation["id"] == conversation_id:
            conversation.update(updates)
            conversation["updated_at"] = _now_iso()
            save_conversations(conversations)
            return conversation

    raise ConversationError(
        message="会话不存在",
        detail=f"conversation_id={conversation_id}",
        status_code=404,
    )


def load_messages() -> list[dict[str, Any]]:
    return _load_json_list(MESSAGES_FILE)


def save_messages(messages: list[dict[str, Any]]) -> None:
    _save_json_list(MESSAGES_FILE, messages)


def create_message(
    conversation_id: str,
    role: str,
    content: str,
    metadata: dict[str, Any] | None = None,
) -> dict[str, Any]:
    if role not in ALLOWED_MESSAGE_ROLES:
        raise ConversationError(
            message="非法消息角色",
            detail=f"role={role}",
            status_code=400,
        )

    message = {
        "id": str(uuid.uuid4()),
        "conversation_id": conversation_id,
        "role": role,
        "content": content,
        "metadata": metadata or {},
        "created_at": _now_iso(),
    }

    messages = load_messages()
    messages.append(message)
    save_messages(messages)

    return message


def list_messages(conversation_id: str) -> list[dict[str, Any]]:
    messages = load_messages()
    return [
        message
        for message in messages
        if message["conversation_id"] == conversation_id
    ]


def list_recent_messages(
    conversation_id: str,
    limit: int = 10,
) -> list[dict[str, Any]]:
    messages = list_messages(conversation_id)
    return messages[-limit:]


def count_messages(conversation_id: str) -> int:
    return len(list_messages(conversation_id))
```


---

## 1. 为什么需要 Service 层

Store 层只处理数据。

Service 层处理业务链路：

```text
校验 conversation 是否存在
保存 user message
读取最近历史
构造 LLM messages
调用 OllamaProvider
保存 assistant message
返回响应数据
```

如果把这些都写在 `routes.py`，路由会变成大杂烩。

你当前 `routes.py` 已经只是调用 `handle_chat()`、`handle_chat_stream()`，这说明你已经在用“路由薄、服务厚”的方向。

---

## 2. 为什么 build_llm_messages 暂时放在 conversation_service.py

第三周的上下文策略很简单：

```text
固定 system prompt
+ 最近 10 条历史消息
+ 当前用户消息
```

所以暂时放在 `conversation_service.py`。

Week 4 会升级成：

```text
ContextBuilder
Summary Memory
Token / 字符数裁剪
context-preview 调试接口
```

也就是你计划里的 Context Window 阶段。

---

## 3. 为什么只取最近 10 条消息

不能把所有历史都塞给模型，原因：

```text
上下文窗口有限
请求会越来越慢
成本会越来越高
旧消息可能变成噪音
模型可能被过期上下文干扰
```

第三周先取最近 10 条，是工程上的安全默认值。

---

## 4. 当前用户消息不能重复加入两次

这是最容易写错的地方。

正确顺序：

```text
1. 先保存 user message
2. build_llm_messages 读取最近消息
3. 如果最近消息已经包含当前 user message，就不要再 append current_content
```

本教程选择更清晰的方式：

```text
先保存 user message
再读取最近 10 条消息
build_llm_messages 只根据已保存消息构造上下文
不额外 append current_content
```

这样不会重复。

```python
from collections.abc import Iterable
from json import dumps
from time import perf_counter
from typing import Any

from src.app.config import get_ollama_model
from src.app.conversation_store import (
    count_messages,
    create_conversation,
    create_message,
    get_conversation,
    list_conversations,
    list_messages,
    list_recent_messages,
    update_conversation,
)
from src.app.exceptions import ConversationError
from src.app.logger import get_logger
from src.app.services.llm.ollama_provider import OllamaProvider

logger = get_logger()

DEFAULT_SYSTEM_PROMPT = "你是一个专业、耐心、严谨的 AI 开发学习助手。请使用简体中文回答。"
DEFAULT_CONTEXT_MESSAGE_LIMIT = 10


def _ensure_conversation_exists(conversation_id: str) -> dict[str, Any]:
    conversation = get_conversation(conversation_id)

    if conversation is None:
        raise ConversationError(
            message="会话不存在",
            detail=f"conversation_id={conversation_id}",
            status_code=404,
        )

    return conversation


def _with_message_count(conversation: dict[str, Any]) -> dict[str, Any]:
    return {
        **conversation,
        "message_count": count_messages(conversation["id"]),
    }


def create_new_conversation(
    title: str,
    model: str | None = None,
) -> dict[str, Any]:
    if not title.strip():
        raise ConversationError(
            message="会话标题不能为空",
            status_code=400,
        )

    return create_conversation(
        title=title.strip(),
        model=model,
    )


def get_conversation_list() -> list[dict[str, Any]]:
    conversations = list_conversations()
    return [_with_message_count(item) for item in conversations]


def get_conversation_detail(conversation_id: str) -> dict[str, Any]:
    conversation = _ensure_conversation_exists(conversation_id)
    return _with_message_count(conversation)


def get_conversation_messages(conversation_id: str) -> list[dict[str, Any]]:
    _ensure_conversation_exists(conversation_id)
    return list_messages(conversation_id)


def build_llm_messages(
    conversation_id: str,
    limit: int = DEFAULT_CONTEXT_MESSAGE_LIMIT,
) -> list[dict[str, str]]:
    conversation = _ensure_conversation_exists(conversation_id)
    recent_messages = list_recent_messages(conversation_id, limit=limit)

    llm_messages: list[dict[str, str]] = [
        {
            "role": "system",
            "content": DEFAULT_SYSTEM_PROMPT,
        }
    ]

    if conversation.get("summary"):
        llm_messages.append(
            {
                "role": "system",
                "content": f"当前会话摘要：{conversation['summary']}",
            }
        )

    for message in recent_messages:
        role = message["role"]

        if role not in {"user", "assistant", "system", "tool"}:
            continue

        llm_messages.append(
            {
                "role": role,
                "content": message["content"],
            }
        )

    return llm_messages


def send_conversation_message(
    conversation_id: str,
    content: str,
    model: str | None = None,
) -> dict[str, Any]:
    conversation = _ensure_conversation_exists(conversation_id)

    if not content.strip():
        raise ConversationError(
            message="消息内容不能为空",
            status_code=400,
        )

    selected_model = model or conversation.get("model") or get_ollama_model()

    user_message = create_message(
        conversation_id=conversation_id,
        role="user",
        content=content.strip(),
        metadata={},
    )

    llm_messages = build_llm_messages(conversation_id)

    provider = OllamaProvider()

    start = perf_counter()
    llm_response = provider.chat(
        messages=llm_messages,
        model=selected_model,
    )
    elapsed = perf_counter() - start
    latency_ms = int(elapsed * 1000)

    assistant_message = create_message(
        conversation_id=conversation_id,
        role="assistant",
        content=llm_response.content,
        metadata={
            "model": llm_response.model,
            "provider": llm_response.provider,
            "latency_ms": latency_ms,
            "context_message_count": len(llm_messages),
            "is_stream": False,
        },
    )

    update_conversation(conversation_id, {})

    logger.info(
        "会话消息处理完成: conversation_id=%s latency_ms=%s",
        conversation_id,
        latency_ms,
    )

    return {
        "user_message": user_message,
        "assistant_message": assistant_message,
    }


def _sse_event(event: str, data: Any) -> str:
    if isinstance(data, str):
        payload = data
    else:
        payload = dumps(data, ensure_ascii=False)

    return f"event: {event}\ndata: {payload}\n\n"


def stream_conversation_message(
    conversation_id: str,
    content: str,
    model: str | None = None,
) -> Iterable[str]:
    conversation = _ensure_conversation_exists(conversation_id)

    if not content.strip():
        yield _sse_event(
            "error",
            {
                "code": "CONVERSATION_ERROR",
                "message": "消息内容不能为空",
            },
        )
        yield _sse_event("done", "[DONE]")
        return

    selected_model = model or conversation.get("model") or get_ollama_model()

    try:
        user_message = create_message(
            conversation_id=conversation_id,
            role="user",
            content=content.strip(),
            metadata={},
        )

        llm_messages = build_llm_messages(conversation_id)

        yield _sse_event(
            "message_start",
            {
                "conversation_id": conversation_id,
                "user_message_id": user_message["id"],
            },
        )

        provider = OllamaProvider()
        full_answer_parts: list[str] = []

        start = perf_counter()

        for chunk in provider.stream_chat(
            messages=llm_messages,
            model=selected_model,
        ):
            if chunk.done:
                break

            full_answer_parts.append(chunk.delta)

            yield _sse_event(
                "delta",
                {
                    "delta": chunk.delta,
                },
            )

        elapsed = perf_counter() - start
        latency_ms = int(elapsed * 1000)
        full_answer = "".join(full_answer_parts)

        assistant_message = create_message(
            conversation_id=conversation_id,
            role="assistant",
            content=full_answer,
            metadata={
                "model": selected_model,
                "provider": "ollama",
                "latency_ms": latency_ms,
                "context_message_count": len(llm_messages),
                "is_stream": True,
            },
        )

        update_conversation(conversation_id, {})

        yield _sse_event(
            "message_end",
            {
                "assistant_message_id": assistant_message["id"],
                "content": full_answer,
                "latency_ms": latency_ms,
            },
        )
        yield _sse_event("done", "[DONE]")

    except Exception as exc:
        logger.exception(
            "流式会话消息失败: conversation_id=%s error=%s",
            conversation_id,
            exc,
        )

        yield _sse_event(
            "error",
            {
                "code": "STREAM_CONVERSATION_MESSAGE_ERROR",
                "message": "流式会话消息失败",
                "detail": str(exc),
            },
        )
        yield _sse_event("done", "[DONE]")
```


每个 delta 都推给前端。


流结束后拼出完整 assistant content。


这时再保存 assistant message。

## 8. 流式 assistant message 入库方案比较

### 方案 A：stream 结束后再创建 assistant message

流程：

```text
开始 stream
  ↓
只保存 user message
  ↓
delta 持续返回
  ↓
服务端拼接 full_answer
  ↓
结束后创建 assistant message
```

优点：

```text
实现简单
不会出现 content 为空的 assistant message
不需要 update_message
适合第三周
```

缺点：

```text
stream 过程中查消息列表，看不到正在生成的 assistant message
如果中途失败，不会保存半截回答
```

### 方案 B：stream 开始前创建空 assistant message，结束后 update

流程：

```text
创建 assistant message，content=""
  ↓
stream 过程中返回 delta
  ↓
结束后 update content
```

优点：

```text
前端可以提前拿到 assistant_message_id
更像生产系统
可支持中途状态：generating / failed / completed
```

缺点：

```text
需要 update_message
状态管理更复杂
初学阶段容易写乱
```

### 第三周选择

本教程选择方案 A：

```text
stream 结束后再创建 assistant message
```

原因：

```text
第三周目标是先跑通 Conversation + Message 主链路
不引入 message status
不增加 update_message
更容易定位问题
```

Week 10 做 Agent Trace 或流式状态时，再升级到方案 B。

# Day 5：实现流式会话消息接口

## 目标

新增接口：

```text
POST /conversations/{conversation_id}/messages/stream
```

要求：

```text
支持 SSE delta
结束后保存完整 assistant message
失败时返回 error event
```

---

## 1. 为什么流式接口不能加 response_model

普通接口返回 JSON：

```json
{
  "user_message": {},
  "assistant_message": {}
}
```

所以可以：

```python
response_model=SendMessageResponse
```

流式接口返回的是连续字符串协议：

```text
event: delta
data: {"delta":"你"}

event: delta
data: {"delta":"好"}
```

不是一次性 JSON，所以不能简单加 `response_model`。

---

## 2. SSE 是什么

SSE 全称：

```text
Server-Sent Events
```

它本质是服务端不断往 HTTP 响应里写字符串。

每条消息格式：

```text
event: delta
data: {"delta":"你好"}

```

注意最后必须有两个换行：

```text
\n\n
```

否则浏览器不知道一条事件结束了。

---

## 3. data-only 和 event 格式比较

### data-only

```text
data: {"delta":"你"}

data: {"done":true}
```

优点：

```text
简单
第二周已经用过
前端只监听 onmessage 即可
```

缺点：

```text
不同事件类型混在一起
前端需要自己判断 done/error/delta
后续 Agent Trace 不够清晰
```

### event 格式

```text
event: message_start
data: {"user_message_id":"xxx"}

event: delta
data: {"delta":"你"}

event: message_end
data: {"assistant_message_id":"xxx"}

event: done
data: [DONE]
```

优点：

```text
事件类型清晰
前端可以分别监听
适合后续 Agent Trace / Tool Call / Sources
```

缺点：

```text
比 data-only 多一点代码
前端要监听自定义事件
```

第三周建议直接使用 `event: xxx`，因为你后续要做 Agent Trace。

```ts
switch (eventName) {
```

# 13. 常见报错排查

## 1. `ConversationError: 会话不存在`

原因：

```text
conversation_id 写错
data/conversations.json 被清空
创建会话后没有复制正确 id
```

检查：

```bash
cat data/conversations.json
```

## 2. `REQUEST_VALIDATION_ERROR`

原因：

```text
请求 body 字段不对
content 为空
title 为空
JSON 格式错误
```

错误示例：

```json
{
  "message": "你好"
}
```

正确：

```json
{
  "content": "你好"
}
```

## 3. 流式接口一次性返回，不是逐字输出

原因：

```text
curl 没有加 -N
代理层缓冲
浏览器端 fetch 读取方式不对
模型本身输出较快
```

测试时使用：

```bash
curl -N ...
```

## 4. 第二轮无法记住第一轮

按顺序排查：

```text
1. data/messages.json 是否有第一轮 user message
2. 是否有第一轮 assistant message
3. list_recent_messages 是否返回历史
4. build_llm_messages 是否包含历史
5. provider.chat 是否收到完整 messages
6. 当前问题是否重复 append
7. 是否错误使用旧 build_chat_prompt 覆盖上下文
```

## 5. JSON 中文乱码

确认写 JSON 时使用：

```python
json.dumps(items, ensure_ascii=False, indent=2)
```

## 6. `metadata` 变成共享对象

不要写：

```python
metadata: dict = {}
```

要写：

```python
metadata: dict[str, Any] = Field(default_factory=dict)
```

## 7. `role` 写错

合法值只能是：

```text
system
user
assistant
tool
```

不要使用：

```text
bot
ai
human
```

# 14. 本周必须掌握的概念

## 1. Conversation

不是简单列表，而是 AI Chat 产品的会话边界。

## 2. Message

不是普通日志，而是模型上下文的原材料。

## 3. role

不是普通字符串，而是 LLM 理解对话结构的关键。

## 4. metadata

不是可有可无的字段，而是后续调试、性能分析、trace、RAG sources、tool calls 的承载点。

## 5. 多轮上下文

不是“把所有历史全塞进去”，而是后续 Context Window 的基础。

## 6. Store 层

只负责数据读写，不关心 FastAPI，也不调用 LLM。

## 7. Service 层

负责业务链路编排。

## 8. SSE

是字符串协议，不是普通 JSON 响应。

## 9. 流式入库

assistant 完整内容要等 stream 结束后才知道。

## 10. `/chat` 和正式会话接口的区别

`/chat` 是调试接口；`/conversations/{id}/messages` 是正式产品接口。

# 15. 第三周完成后的面试表达

## 1 分钟版本

```text
我在这个项目里把原来的单轮 Chat Demo 升级成了正式的多会话 AI Chat Backend。后端增加了 Conversation 和 Message 两层数据结构，支持创建会话、查询会话列表、查询消息列表、在指定会话中发送普通消息和流式消息。每次发送消息时，系统会先保存 user message，再读取该会话最近 10 条历史消息构造 LLM messages，调用 OllamaProvider 生成回答，最后保存 assistant message。流式接口会通过 SSE 返回 delta，并在 stream 结束后把完整 assistant message 保存下来。这个设计为后续 Context Window、Summary Memory、RAG 和 Agent Trace 打下了基础。
```

## 3 分钟版本

```text
原项目只有 /chat 和 /chat/stream，本质是单轮模型调用，虽然可以调 Ollama，但无法支撑真实 AI Chat 产品。第三周我新增了 Conversation 和 Message 模型。Conversation 表示一次连续聊天的边界，Message 表示会话中的每条消息，并且包含 role、content、metadata、created_at 等字段。

在业务流程上，我把数据读写放到 conversation_store.py，把业务编排放到 conversation_service.py，路由层只负责 HTTP 输入输出。普通消息接口会校验 conversation，保存 user message，读取最近 10 条历史消息，构造标准 LLM messages，然后调用 OllamaProvider.chat，最后保存 assistant message。流式接口则先保存 user message，边 stream 边返回 SSE delta，同时在服务端累积完整回答，结束后保存 assistant message。

这个版本暂时使用本地 JSON 文件模拟 conversations 和 messages 两张表，不引入 PostgreSQL 和 SQLAlchemy，目的是先把会话系统的数据模型和调用链路跑通。下一步会把 build_llm_messages 升级为 ContextBuilder，并加入 Summary Memory，解决上下文无限增长的问题。
```

## 5 分钟版本结构

```text
1. 背景：原来只有单轮 /chat，不是正式 AI Chat 产品
2. 数据模型：Conversation 是会话边界，Message 是上下文原材料
3. role 设计：system/user/assistant/tool 为后续 Agent 做准备
4. 分层设计：routes -> service -> store -> provider
5. 普通消息链路：保存 user -> 构造上下文 -> 调模型 -> 保存 assistant
6. 流式消息链路：SSE delta -> full_answer_parts -> 结束后入库
7. 存储策略：第三周 JSON，未来迁移 PostgreSQL
8. 限制：只取最近 10 条，不做 summary，不做复杂 token 裁剪
9. 下一步：ContextBuilder + Summary Memory + RAG
```
