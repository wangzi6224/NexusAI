## 2.1 修改 `src/app/schemas.py`

把聊天相关模型改成下面这样。

```python
from pydantic import BaseModel, Field


class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1, description="用户输入的问题")
    model: str | None = Field(default=None, description="可选，指定本次请求使用的模型")


class TokenUsage(BaseModel):
    prompt_tokens: int = 0
    completion_tokens: int = 0
    total_tokens: int = 0


class ChatResponse(BaseModel):
    answer: str
    provider: str
    model: str
    latency_ms: int
    usage: TokenUsage = Field(default_factory=TokenUsage)


class HealthResponse(BaseModel):
    ok: bool
    message: str


class HistoryItem(BaseModel):
    timestamp: str
    model: str
    user_input: str
    prompt: str
    answer: str
    elapsed_seconds: float


class ClearHistoryResponse(BaseModel):
    success: bool
    message: str


class ModelsResponse(BaseModel):
    current_model: str
    available_models: list[str]


class SelectModelRequest(BaseModel):
    model: str


class SelectModelResponse(BaseModel):
    success: bool
    selected_model: str
    message: str
```

---

## 2.2 概念解释：Pydantic 是什么？

Pydantic 是 Python 里非常重要的数据校验库。

在 FastAPI 里，它主要做三件事：

```text
1. 校验请求体
2. 生成接口文档
3. 保证响应结构稳定
```

比如：

```python
class ChatRequest(BaseModel):
    message: str
```

意思是：

```text
请求体必须是 JSON
JSON 里必须有 message
message 必须是字符串
```

如果用户传：

```json
{
  "message": 123
}
```

FastAPI 会自动返回参数错误，不需要你手动判断。

---

## 2.3 概念解释：`Field(...)` 是什么？

```python
message: str = Field(..., min_length=1, description="用户输入的问题")
```

这里的 `...` 表示必填。

等价理解：

```text
message 必须传
最小长度是 1
description 会出现在 Swagger 文档里
```

类比 zod：

```ts
const ChatRequest = z.object({
  message: z.string().min(1)
})
```

---

## 2.4 概念解释：`default_factory` 是什么？

```python
usage: TokenUsage = Field(default_factory=TokenUsage)
```

意思是：

```text
如果没有传 usage，就自动创建一个新的 TokenUsage()
```

不要写成：

```python
usage: TokenUsage = TokenUsage()
```

因为对象默认值可能被复用。`default_factory` 更安全。

## 3.2 新建 `src/app/services/llm/base.py`

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass


@dataclass
class LLMUsage:
    prompt_tokens: int = 0
    completion_tokens: int = 0
    total_tokens: int = 0


@dataclass
class LLMResponse:
    content: str
    provider: str
    model: str
    usage: LLMUsage


class LLMProvider(ABC):
    name: str

    @abstractmethod
    def chat(self, messages: list[dict[str, str]], model: str | None = None) -> LLMResponse:
        """调用聊天模型并返回统一结构。"""
        raise NotImplementedError
```

---

## 3.3 概念解释：为什么需要 Provider？

Provider 本质是“适配器”。

你的业务代码不应该关心底层是：

```text
Ollama
OpenAI
vLLM
DashScope
DeepSeek API
Claude API
```

业务代码只应该关心：

```python
provider.chat(messages)
```

也就是说：

```text
业务层调用统一接口
底层 Provider 负责适配不同模型服务
```

类比前端：

```ts
interface StorageProvider {
  get(key: string): string
  set(key: string, value: string): void
}

class LocalStorageProvider implements StorageProvider {}
class RedisStorageProvider implements StorageProvider {}
```

你不会让业务代码到处直接写 `localStorage.getItem()`，因为以后换实现会很痛苦。

---

## 3.4 概念解释：`ABC` 和 `abstractmethod`

```python
from abc import ABC, abstractmethod
```

`ABC` 是 Abstract Base Class，抽象基类。

```python
class LLMProvider(ABC):
```

意思是：

```text
LLMProvider 不是直接拿来实例化的
它是给子类继承的接口规范
```

```python
@abstractmethod
def chat(...):
```

意思是：

```text
所有子类必须实现 chat 方法
否则不能实例化
```

类比 TypeScript：

```ts
interface LLMProvider {
  chat(messages: Message[]): Promise<LLMResponse>
}
```

Python 没有 TS 那种 interface，所以常用 `ABC + abstractmethod` 表达接口约束。

---

## 3.5 概念解释：`dataclass`

```python
@dataclass
class LLMResponse:
    content: str
    provider: str
    model: str
    usage: LLMUsage
```

`dataclass` 主要用于定义“数据对象”。

它会自动生成：

```text
__init__
__repr__
__eq__
```

你可以直接这样创建：

```python
response = LLMResponse(
    content="你好",
    provider="ollama",
    model="qwen2.5:7b",
    usage=LLMUsage(),
)
```

类比 TypeScript：

```ts
type LLMResponse = {
  content: string
  provider: string
  model: string
  usage: LLMUsage
}
```

# 4. 第四天：实现 OllamaProvider

## 4.1 新建 `src/app/services/llm/ollama_provider.py`

```python
import requests

from src.app.config import (
    get_ollama_base_url,
    get_ollama_keep_alive,
    get_ollama_timeout,
)
from src.app.runtime_config import get_selected_model
from src.app.services.llm.base import LLMProvider, LLMResponse, LLMUsage


class OllamaProvider(LLMProvider):
    name = "ollama"

    def chat(self, messages: list[dict[str, str]], model: str | None = None) -> LLMResponse:
        base_url = get_ollama_base_url()
        selected_model = model or get_selected_model()
        timeout = get_ollama_timeout()
        keep_alive = get_ollama_keep_alive()

        url = f"{base_url}/api/chat"

        payload = {
            "model": selected_model,
            "messages": messages,
            "stream": False,
            "keep_alive": keep_alive,
        }

        try:
            response = requests.post(url, json=payload, timeout=timeout)
            response.raise_for_status()
        except requests.exceptions.Timeout as exc:
            raise RuntimeError("请求 Ollama 超时，请检查模型是否正在加载，或机器性能是否不足") from exc
        except requests.exceptions.HTTPError as exc:
            raise RuntimeError(
                f"Ollama HTTP 请求失败，状态码: {response.status_code}，响应内容: {response.text}"
            ) from exc
        except requests.exceptions.RequestException as exc:
            raise RuntimeError(f"请求 Ollama 失败: {exc}") from exc

        data = response.json()

        try:
            content = data["message"]["content"]
        except (KeyError, TypeError) as exc:
            raise RuntimeError(f"Ollama 返回结构异常: {data}") from exc

        usage = LLMUsage(
            prompt_tokens=data.get("prompt_eval_count", 0) or 0,
            completion_tokens=data.get("eval_count", 0) or 0,
            total_tokens=(data.get("prompt_eval_count", 0) or 0) + (data.get("eval_count", 0) or 0),
        )

        return LLMResponse(
            content=content,
            provider=self.name,
            model=selected_model,
            usage=usage,
        )
```

---

## 4.2 概念解释：Ollama `/api/chat` 请求体

Ollama 的聊天接口核心结构是：

```json
{
  "model": "qwen2.5:7b",
  "messages": [
    {
      "role": "user",
      "content": "什么是 RAG？"
    }
  ],
  "stream": false
}
```

### `model`

表示你要调用哪个模型。

```json
"model": "qwen2.5:7b"
```

这要求你本地已经有这个模型：

```bash
ollama list
```

---

### `messages`

这是聊天上下文。

```json
"messages": [
  {
    "role": "user",
    "content": "什么是 RAG？"
  }
]
```

常见角色：

| role        | 含义                 |
| ----------- | ------------------ |
| `system`    | 系统指令，定义模型身份和规则     |
| `user`      | 用户消息               |
| `assistant` | 模型之前的回答            |
| `tool`      | 工具调用结果，后面 Agent 会用 |

第一周先只用 `user`。

后面做 RAG 时会变成：

```python
messages = [
    {
        "role": "system",
        "content": "你是企业研发规范助手，只能根据资料回答。"
    },
    {
        "role": "user",
        "content": "根据以下资料回答问题..."
    }
]
```

---

### `stream`

```json
"stream": false
```

表示一次性返回完整答案。

如果是：

```json
"stream": true
```

Ollama 会一段一段返回 token。这个放到第二周做。

---

### `keep_alive`

```json
"keep_alive": "5m"
```

表示模型加载后保留多久。

大模型首次加载慢，`keep_alive` 可以减少反复加载模型的成本。

# 5. 第五天：改造 chat_service

现在 `chat_service.py` 直接调用 `chat_completion(prompt)`，我们改成调用 `OllamaProvider`。

## 5.1 修改 `src/app/services/chat_service.py`

```python
from datetime import datetime
from time import perf_counter

from src.app.config import get_ollama_model
from src.app.history import append_history
from src.app.logger import get_logger
from src.app.prompts import build_chat_prompt
from src.app.schemas import ChatResponse, TokenUsage
from src.app.services.llm.ollama_provider import OllamaProvider

logger = get_logger()


def handle_chat(message: str, model: str | None = None) -> ChatResponse:
    selected_model = model or get_ollama_model()
    logger.info("收到聊天请求，provider=ollama, model=%s", selected_model)

    prompt = build_chat_prompt(message)

    messages = [
        {
            "role": "user",
            "content": prompt,
        }
    ]

    provider = OllamaProvider()

    start = perf_counter()
    llm_response = provider.chat(messages=messages, model=selected_model)
    elapsed = perf_counter() - start
    latency_ms = int(elapsed * 1000)

    append_history(
        {
            "timestamp": datetime.now().isoformat(),
            "model": llm_response.model,
            "user_input": message,
            "prompt": prompt,
            "answer": llm_response.content,
            "elapsed_seconds": round(elapsed, 2),
        }
    )

    logger.info("聊天请求处理完成，耗时 %s ms", latency_ms)

    return ChatResponse(
        answer=llm_response.content,
        provider=llm_response.provider,
        model=llm_response.model,
        latency_ms=latency_ms,
        usage=TokenUsage(
            prompt_tokens=llm_response.usage.prompt_tokens,
            completion_tokens=llm_response.usage.completion_tokens,
            total_tokens=llm_response.usage.total_tokens,
        ),
    )
```

## 5.2 概念解释：service 层是什么？

现在你的项目大致分层是：

```text
routes.py
  ↓
chat_service.py
  ↓
OllamaProvider
  ↓
Ollama HTTP API
```

职责应该这样分：

| 层          | 文件                   | 职责                    |
| ---------- | -------------------- | --------------------- |
| 路由层        | `routes.py`          | 接收 HTTP 请求，返回 HTTP 响应 |
| 业务层        | `chat_service.py`    | 编排业务流程、计时、记录历史        |
| Provider 层 | `ollama_provider.py` | 适配模型服务                |
| 配置层        | `config.py`          | 读取环境变量                |
| Schema 层   | `schemas.py`         | 定义请求和响应结构             |

不要让 `routes.py` 直接调用 Ollama。

原因是：

```text
路由层应该薄
业务逻辑应该放 service
外部系统调用应该放 provider/client
```

这和你熟悉的前端/Node 后端结构类似：

```text
Controller -> Service -> SDK Client
```

```python
def chat(request: ChatRequest) -> ChatResponse:
    try:
        return handle_chat(message=request.message, model=request.model)
    except Exception as exc:
        raise HTTPException(status_code=500, detail=f"模型调用失败: {exc}") from exc
```

---

## 6.2 概念解释：`response_model=ChatResponse`

```python
@router.post("/chat", response_model=ChatResponse)
```

意思是：

```text
这个接口的返回值必须符合 ChatResponse 结构
```

好处：

```text
1. 自动生成 Swagger 文档
2. 返回字段受控
3. 不会把多余字段乱返回给前端
4. 接口契约更稳定
```

类比前端：

```ts
type ChatResponse = {
  answer: string
  provider: string
  model: string
  latency_ms: number
}
```

---

## 6.3 概念解释：为什么 `HTTPException`

```python
raise HTTPException(status_code=500, detail=f"模型调用失败: {exc}") from exc
```

这是 FastAPI 用来返回 HTTP 错误的方式。

比如 Ollama 没启动时，普通 Python 异常如果不处理，会返回一个很难看的服务端错误。

用 `HTTPException` 后，前端收到的是标准 JSON：

```json
{
  "detail": "模型调用失败: 请求 Ollama 失败..."
}
```

后续第二周我们会把它升级成统一错误结构：

```json
{
  "error": {
    "code": "LLM_PROVIDER_ERROR",
    "message": "Ollama service unavailable"
  }
}
```

# 8. 第一周你必须理解的核心概念

## 8.1 FastAPI

FastAPI 是 Python Web 框架。

你可以把它理解成：

```text
Python 版 NestJS / Express
```

最小接口：

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/hello")
def hello():
    return {"message": "hello"}
```

你的项目里没有直接在 `server.py` 写很多接口，而是：

```python
app.include_router(router)
```

这表示把 `routes.py` 里的路由挂进主应用。

---

## 8.2 APIRouter

`APIRouter` 是 FastAPI 的路由模块化工具。

```python
router = APIRouter()
```

然后：

```python
@router.post("/chat")
def chat():
    ...
```

最后在 `server.py`：

```python
app.include_router(router)
```

类比 NestJS：

```text
Controller 被 Module 注册
```

类比 Express：

```ts
const router = express.Router()
app.use(router)
```

---

## 8.3 Pydantic

Pydantic 是数据模型和校验工具。

在 AI 项目里它非常重要，因为 AI 工程经常需要结构化：

```text
请求结构
响应结构
工具参数
工具返回值
Agent 状态
RAG sources
评测 case
```

后面你做 Agent tool calling 时，会反复遇到：

```text
schema
json schema
zod
pydantic
```

它们的核心目标都是：

```text
让数据结构可描述、可校验、可约束
```

---

## 8.4 Ollama

Ollama 是本地模型运行工具。

你的 Python 服务并不直接“加载模型权重”，而是：

```text
Python FastAPI
  ↓ HTTP 请求
Ollama 服务
  ↓
本地模型
```

所以你的程序调用模型，本质是调用 HTTP API：

```text
POST http://127.0.0.1:11434/api/chat
```

这点非常关键。

你不是在 Python 里直接运行大模型，而是在 Python 里调用一个模型服务。

---

## 8.5 Provider 抽象

Provider 是为了隔离模型供应商差异。

现在：

```text
OllamaProvider
```

以后可以加：

```text
OpenAIProvider
VLLMProvider
DashScopeProvider
DeepSeekProvider
```

业务层不变：

```python
provider.chat(messages)
```

只有 Provider 内部实现不同。

这是 AI 应用工程最基础的可扩展设计。

---

## 8.6 messages 格式

大模型聊天接口通常不是简单传一个字符串，而是传消息数组：

```python
messages = [
    {
        "role": "user",
        "content": "什么是 RAG？"
    }
]
```

后面会扩展成：

```python
messages = [
    {
        "role": "system",
        "content": "你是企业研发规范助手。"
    },
    {
        "role": "user",
        "content": "根据资料回答问题..."
    }
]
```

再往后 Agent 会变成：

```python
messages = [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."},
    {"role": "tool", "content": "..."}
]
```

所以第一周你要提前熟悉 `messages`，不要一直停留在 `prompt: str`。

---

## 8.7 latency

```python
start = perf_counter()
...
elapsed = perf_counter() - start
latency_ms = int(elapsed * 1000)
```

`latency_ms` 是请求耗时，单位毫秒。

AI 应用必须记录耗时，因为：

```text
模型加载慢
首 token 慢
生成速度慢
RAG 检索慢
数据库慢
```

这些都要靠耗时定位。

---

## 8.8 usage

```json
{
  "prompt_tokens": 0,
  "completion_tokens": 0,
  "total_tokens": 0
}
```

`usage` 是 token 使用量。

虽然 Ollama 不一定像 OpenAI 那样稳定返回完整 usage，但你要先把接口结构设计好。

后续接商业模型时，usage 很重要：

```text
计费
限流
分析成本
优化 prompt
统计用户消耗
```
